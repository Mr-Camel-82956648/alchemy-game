# AI法阵·炼金术士 Roadmap

## 1. 项目目标

本项目是一个 AI 原生游戏原型。玩家通过自然语言输入或合成已有法阵，生成新的法阵大招，并在战斗中使用它们对抗不同属性的敌人。

当前目标不是做完整商业化产品，而是先跑通以下核心闭环：

1. 玩家输入 / 合成
2. 后端生成结构化法阵结果
3. 前端接收并管理法阵
4. 法阵主属性进入战斗伤害判定
5. 怪物根据视觉特征配置免疫属性
6. 玩家通过合成与属性克制解决战斗问题

---

## 2. 当前采用的工程结构

采用单仓库双项目结构：

```txt
alchemy-game/
  README.md
  docs/
  frontend/
  backend/
```

### frontend
负责：
- 游戏主循环
- 战斗逻辑
- 法阵展示
- 怪物展示
- 局内法阵管理
- 命中与伤害结算
- 调用后端 forge 接口

### backend
负责：
- 接收法阵合成请求
- 融合输入提示词
- 输出视觉描述
- 解析主属性/副属性
- 返回结构化法阵结果
- 后续可接入真实 AI 生图/生视频服务

## 4. 当前核心设计原则

### 4.1 视觉优先
玩家最终看到的视觉表现必须和数值结果一致。

### 4.2 统一真相源
属性解析模型面对的不是玩家原始输入，而是融合后的“视觉描述”。

链路如下：

1. 输入 A + 输入 B
2. 融合模型输出视觉描述
3. 生图/生视频与属性解析都基于同一段视觉描述
4. 前端消费结构化法阵结果

### 4.3 规则极简
首版只做最小可玩规则，不上复杂元素连锁与抗性百分比。

## 5. 数值规则（当前冻结版本）

### 5.1 元素体系
四属性固定：
- 火
- 冰
- 雷
- 蚀

### 5.2 法阵字段
统一法阵结构至少包含：

- id
- name
- status
- generation
- visualDesc
- mainAttr
- subAttr
- baseAtk
- videoUrl（可选）
- imageUrl（可选）
- promptA / promptB（可选）
- legacyIndex（兼容旧前端，可选）

### 5.3 基础攻击力
公式固定：

`baseAtk = 100 * (1 + 0.3 * (generation - 1))`

例：
- 1代 = 100
- 2代 = 130
- 3代 = 160
- 4代 = 190
- 5代 = 220

### 5.4 普通怪规则
- 只有 1 个免疫属性
- 若法阵主属性撞免疫，则伤害为 0
- 否则造成完整基础伤害

### 5.5 Boss 规则
- 免疫 2 个属性
- 可破盾属性 = 四属性中剩余两个属性
- 只有法阵主属性命中破盾属性时才造成完整伤害，否则为 0

### 5.6 副属性规则
- 当前只记录
- 不进入战斗结算
- 作为未来扩展接口保留

## 6. 法阵迁移策略

当前法阵分为三类：

### 6.1 legacy 法阵
特点：
- 只有 MP4 或旧索引
- 无提示词
- 无属性
- 无完整结构化数据

处理方式：
- 纳入统一注册表
- 标记 `status = legacy`
- 保留 `legacyIndex`
- 不要求立即可参与完整新战斗系统

### 6.2 partial 法阵
特点：
- 视觉已可用
- 有提示词来源
- 尚未完全结构化
- 尚未接 AI 生成链路

处理方式：
- 优先迁移
- 人工补齐最小结构：
  - name
  - promptA / promptB
  - generation
  - mainAttr / subAttr
  - baseAtk

### 6.3 complete 法阵
特点：
- 来自 AI 生成链路
- 结构完整
- 可直接进入新战斗系统

---

## 7. 怪物迁移策略

当前怪物素材已有视觉与动画，但无属性系统。

处理方式：
1. 保持怪物表现逻辑不动
2. 新增怪物定义表
3. 每个怪物通过结构化字段绑定：
   - id
   - name
   - type
   - hp
   - immuneAttrs
   - assetKey

当前阶段怪物属性先人工配置，不依赖 LLM。

## 8. 前后端分工（冻结）

### 前端负责
- 局内法阵列表
- 怪物和战斗实时逻辑
- 伤害计算调用
- UI 展示
- 调用后端 API 获取法阵生成结果

### 后端负责
- prompt 融合
- 视觉描述生成
- 主副属性解析
- 生成法阵结构化结果
- 返回 imageUrl / videoUrl（首版可 mock）

---

## 9. 当前优先级

### Phase 1：统一数据入口
1. 找到当前前端法阵入口
2. 建立 `Spell` 类型
3. 建立法阵注册表
4. 建立怪物定义表
5. 建立战斗计算函数

### Phase 2：兼容旧前端
1. 让现有法阵展示逻辑改从注册表读取
2. 兼容 legacy 法阵
3. 让部分 partial 法阵可参与新战斗逻辑

### Phase 3：最小后端
1. 建 FastAPI
2. 提供 forge 接口
3. 用 mock 返回法阵对象
4. 前端接入接口并轮询

### Phase 4：逐步完善
1. 补更多法阵结构化数据
2. 接入真实 AI 模型
3. 再考虑更复杂扩展

## 10. 当前明确不做的事情

以下内容当前阶段不做：

1. 云端账号与登录
2. 云存档
3. 数据库
4. 永久法阵收藏库

7. 大规模资源重命名
8. 把所有 legacy 法阵一次性重做
9. 复杂元素连锁机制

## 11. 资源管理原则

### 法阵资源
- 一律通过稳定 id 管理
- 不通过数组位置管理素材
- 删除法阵时优先删除注册表引用，不依赖数组下标

### 怪物资源
- 表现资源与属性数据解耦
- 通过 `assetKey` 关联表现资源

### 素材删除原则
- 运行时删除通常只删索引或注册表引用
- 开发阶段删物理文件时，要同步清理引用字段
- 严禁以数组顺序作为资源唯一定位方式
